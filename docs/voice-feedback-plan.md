## 新機能: 音声フィードバック

### コンセプト

**「喋るだけでフィードバックが完了する」**

テキスト選択もクリックも不要。マイクボタンを押して喋れば、AIが内容を解析し、図解の適切な箇所にピンコメントを自動配置する。

### なぜこれが勝てるか

1. **手間の削減レベルが段違い**
   - テキスト選択コメント: 3ステップ（場所選ぶ → クリック → 書く）
   - ワンクリックピン: 2ステップ（クリック → 書く）
   - 音声: 1ステップ（喋る）— 場所指定もテキスト入力もAIが代行

2. **デモのインパクト**
   - 「じゃあフィードバックしてみます」→ 20秒喋る → ピンが自動配置
   - 「テキスト1文字も打ってません」— パンチラインが明快

3. **実用性が高い**
   - 上長は忙しい。喋る方がテキストを書くより圧倒的に楽
   - 会議中に画面共有しながら口頭でFBする場面をデジタル化

4. **業務効率化ツールと同じ土俵に立てる**
   - 「確認作業の時間を半分にする」と言い切れる
   - 議事録自動化 = 「会議後の書き起こしを消す」
   - 音声FB = 「フィードバックの入力作業を消す」

### ユーザー操作フロー

```
① 図解を開く
② マイクボタン（🎙）を押す
③ 喋る:
   「上のセクション、認証と認可の説明、もっとシンプルにして。
    あと真ん中のフロー図は矢印の順番が逆だと思う。
    最後のまとめはこれでOKです。」
④ 停止ボタンを押す（or 無音で自動停止）
⑤ AIが解析中...（数秒）
⑥ ピンが一気に配置される:
   - セクション「認証と認可」にピン 🔴「もっとシンプルに」
   - セクション「フロー図」にピン 🔴「矢印の順番が逆」
   - セクション「まとめ」にピン 🟢「OK」
⑦ 完了。作成者に通知が飛ぶ（Slack連携時）
```

## 技術アーキテクチャ

```
[ブラウザ]
  │
  ├─ Web Speech API（音声→テキスト変換）
  │   or Whisper API（精度重視の場合）
  │
  ├─ 図解HTMLのセクション構造を自動解析
  │   → 各セクションの見出し + 位置座標を抽出
  │
  └─ LLM API（GPT-4o 推奨）
      入力: 音声テキスト + セクション一覧
      出力: [{section: "認証と認可", comment: "もっとシンプルに", priority: "must"}, ...]
      │
      └─ 返ってきた結果をピンとして配置（pin.ts の既存機能を利用）
```

### コンポーネント構成

```
src/widget/
├── index.ts          ← voice.ts を追加でインポート
├── pin.ts            ← 既存。ピン配置・描画（音声からも呼び出す）
├── voice.ts          ← ★新規。音声録音・解析・ピン振り分け
├── state.ts          ← voiceRecording: boolean 等を追加
├── render/
│   └── toggle.ts     ← マイクボタンを追加
└── styles.ts         ← マイク関連CSS追加
```

### 新規ファイル: `src/widget/voice.ts`

主要関数:
- `startRecording()` — 音声録音開始（Web Speech API or MediaRecorder）
- `stopRecording()` — 停止、テキスト取得
- `analyzeAndPlacePins(transcript: string)` — LLM に送信、結果をピンに変換
- `extractSections()` — ページのセクション構造（見出し + 座標）を抽出

### LLM プロンプト設計

```
あなたはフィードバック解析AIです。
以下の図解のセクション一覧と、ユーザーの音声フィードバックを照合し、
各コメントがどのセクションに対するものかを判定してください。

【セクション一覧】
1. "HTTPリクエストの流れ"（見出し、y=0px）
2. "ブラウザがリクエストを送る"（ステップ1、y=200px）
3. "サーバーが処理する"（ステップ2、y=350px）
4. "レスポンスが返る"（ステップ3、y=500px）
5. "ブラウザが表示する"（ステップ4、y=650px）

【音声テキスト】
「上のセクション、認証と認可の説明、もっとシンプルにして。...」

【出力形式】JSON配列で返してください:
[
  {"section_index": 2, "comment": "もっとシンプルに", "priority": "must"},
  ...
]
```

### API キー管理

- widget.js に直接埋め込みは避ける
- 既存の Next.js API Route に `/api/voice-analyze` エンドポイントを追加
- widget → Vercel API → OpenAI の流れでプロキシ
- トークン認証は既存の仕組みを流用

## 実装フェーズ

### Phase 1: 音声→テキスト（2日）

- Web Speech API での音声認識実装
- マイクボタンUI + 録音中インジケーター
- テキスト結果の表示確認

### Phase 2: セクション抽出（1日）

- ページの `<h1>`, `<h2>`, `<h3>`, `.rounded-xl` 等からセクション構造を自動抽出
- 各セクションの見出しテキスト + Y座標を配列化

### Phase 3: LLM 振り分け（2-3日）

- `/api/voice-analyze` エンドポイント作成
- プロンプト設計 + テスト
- レスポンスをパースしてピン配置

### Phase 4: 統合 + UX磨き込み（2-3日）

- 録音→解析→配置の一連のフロー統合
- ローディング演出（ピンが1つずつ配置されるアニメーション）
- エラーハンドリング（音声認識失敗、LLM応答遅延）
- 結果の確認UI（「この内容で合っていますか？」の確認ステップ、任意）

### Phase 5: Slack連携（2日）

- 音声FBが完了 → Slack に「3件のフィードバックが付きました」通知
- Incoming Webhook 方式（Slack App不要）

### Phase 6: デモ準備（2日）

- デモ用図解の準備
- デモシナリオの台本作成
- エッジケース対策（音声が短すぎる、セクション特定できない場合のフォールバック）
